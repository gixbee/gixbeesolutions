Gixbee — Platform Overview Gixbee is an Intent-Based multi-service platform that unifies event planning, gig hiring, job discovery, and vendor listing under one application. Unlike role-based systems, Gixbee lets a single user book a hall, work as a driver, and list their own business — all in the same session. 1.1 Entry Architecture After Login → OTP verification → Name registration, the user is presented with four intent-based entry points: Book Services Find halls, catering, decoration, photography, rentals & instant help workers Find a Job Talent discovery — apply for jobs, attend interviews, track status Earn by Working Live Worker Engine — offer skills, get real-time gig work, earn hourly List My Business Register service, hiring, or rental businesses; manage units & operators Design Principle Gixbee is a facilitator — it never assigns workers or vendors. Every action (accept, confirm, approve) is a conscious choice by the user. This is the Safe Harbor structure across all modules. 2. Book Services Module This is the largest customer-facing module. It splits into two distinct flows after the user selects the event/service location: Plan Services Calendar-based booking for Hall, Catering, Decoration, Photography, Rental. Location-driven (event location, not user location). Instant Help Real-time gig worker dispatch for Electrician, Driver, Cleaning, Nurse. Uses the Live Worker Engine. 2.1 Plan Services — Booking Flow Complete step-by-step flow with code requirements: Step Action Description / Code Requirement 1 Event Location User selects WHERE the event is (not their current location). System filters all services to that district/city. Code: store selectedEventLocation in session/state. Filter service queries by this location. 2 Service Type User picks Hall / Catering / Decoration / Photography / Rental. Code: route to corresponding service listing screen with location param. 3 View Listing Show vendor cards with photos, capacity, description. Code: GET /services?type={type}&location={location}. Display availability badge. 4 Calendar View Tap 'Check Availability' opens calendar. Slots shown as: Available (green), Pending (yellow), Booked (red). Code: GET /calendar/{vendorId} — return blocked dates array. 5 Booking Type User selects Package (fixed price) OR Custom (flexible requirements). Two separate request flows must be implemented. 6 (Pkg) Send Request State set to REQUESTED. Vendor gets push notification. Code: POST /bookings {type:'package', vendorId, date, packageId}. Status: REQUESTED. 6 (Cust) Custom Request User fills event type, guest count, special needs. Code: POST /bookings {type:'custom', vendorId, date, details:{...}}. Status: CUSTOM_REQUESTED. 7 Vendor Decision Vendor Approves or Rejects. If custom: vendor sends quote + notes. Code: PATCH /bookings/{id}/status {action:'approve'|'reject', quote?, notes?}. 8 User Confirms User reviews quote (if custom) and confirms. State: CONFIRMED. Code: PATCH /bookings/{id}/confirm. Calendar date is then blocked. 9 Calendar Block Once CONFIRMED, date blocked for that vendor unit. Code: Insert date into blocked_dates for vendorId. Reflect immediately on calendar. 2.2 Instant Help — Live Worker Flow Step Action Description / Code Requirement 1 Service Location User picks Current Location OR Another Location. If another: enter city/address. Code: store serviceLocation; if GPS use device coords, else geocode input. 2 Presence Check System asks: 'Who will be present at the location?' User selects Self OR Someone Else. If someone else: collect Name, Relation, Phone — this becomes On-Site Contact. Code: store onSiteContact in booking record. 3 Worker Search System finds available workers within radius for selected skill. Code: GET /workers?skill={skill}&lat={lat}&lng={lng}&radius=10km. Sort by distance & rating. 4 Notify Workers System sends job request to multiple workers simultaneously. Code: Push notification via FCM to matched worker devices. Timer: 90 seconds to accept. 5 Worker Accepts First accepting worker gets the job. Code: PATCH /jobs/{id}/accept — set workerId, status=ACCEPTED. Reject others. 6 Customer Confirms Customer sees worker profile and confirms. Code: PATCH /jobs/{id}/confirm — status=CONFIRMED. Movement monitoring starts. 7 Movement Monitor 7-min reminder sent to worker. At 10 min, system checks GPS movement. No movement detected: auto-cancel + strike added. Code: Background job / cron + GPS polling API. 8 Arrival OTP Worker taps 'Arrived'. OTP sent to Customer or On-Site Contact. Worker enters OTP to start work. Code: Generate 4-digit OTP, store with 5-min TTL, verify before marking STARTED. 9 Work Completion Worker taps 'Finish'. Completion OTP sent. Customer ticks 'Work Completed'. Code: Second OTP flow — verify, then set status=COMPLETED. 10 Payment Customer pays. Minimum: 1 hour billing. Code: Calculate hours (ceil to 1hr minimum), show amount, process payment gateway. 11 Rs.12 Activation After first job: worker receives Rs.12 welcome bonus. Must maintain Rs.12 wallet balance to go live again. Code: Check wallet_balance >= 12 before allowing Active toggle. Deduct on each new job accept. Current Flow Issue — Book Services The system currently does not clearly separate 'event location' from 'user current location'. Both Plan Services and Instant Help need distinct location capture flows. Additionally, the Presence Check / On-Site Contact screen is not yet implemented — this is critical for the Arrival OTP to be sent to the correct person. 3. Rental Service (Inside Plan Services) Rental is a sub-category under Plan Services — NOT under Instant Help. It involves delivery logistics, so it requires calendar-based planning, not real-time dispatch. Step Action Description / Code Requirement 1 Select Location User sets event/rental delivery location. Same location picker as Plan Services. 2 Browse Items User sees categories: Generator, Mic Set, Cooler, Pressure Washer, etc. Code: GET /rentals?location={location}&category={category}. 3 View Item Photos, specs, hourly rate, minimum hours shown. Code: GET /rentals/{itemId}. Display pricing: Rs.X/hour, min Y hours. 4 Check Calendar Calendar shows Available / Booked days. Code: GET /rentals/{itemId}/calendar — return blocked dates. 5 Send Request User taps Request Rental with optional note. Status: REQUESTED. Code: POST /rental-bookings {itemId, date, note?}. 6 Vendor Approval Vendor approves or rejects. Code: PATCH /rental-bookings/{id}/status. 7 User Confirms User confirms. Status: CONFIRMED. Entire day blocked in calendar. Code: PATCH /rental-bookings/{id}/confirm. Block full day in item calendar. 8 Coordination After confirmation, customer and vendor coordinate directly (no OTP). Payment: Offline / GPay. Code: No further system intervention — only calendar block needed. Pricing Rule Minimum hours apply. Use 3hrs on 4hr minimum → pay 4hrs. Use 6hrs → pay 6hrs. Implement as: max(actual_hours, min_hours) * hourly_rate. Calendar Logic Entire day blocked on confirmation (to avoid return-delay conflicts). No partial-day slots for rental items. No OTP Rental does not use Arrival/Completion OTP. Coordination is direct between parties after confirmation. 4. Find a Job Module The Talent Discovery Engine. Connects students, freshers, and experienced professionals directly with hiring companies — no agencies, no newspaper ads. Step Action Description / Code Requirement 1 Talent Profile User fills Education, Skills, Experience. Code: POST /talent/profile {education[], skills[], experience[]}. Store in talent_profiles table. 2 Current Status User selects: Final Year / Graduate / Experienced. Code: Store status field. Used for employer filtering. 3 Job Preferences User adds preferred roles (Nurse, Telecaller, etc.) and preferred work locations. Code: Store preferred_roles[], preferred_locations[]. 4 Job Alerts User enables alert toggle. Code: Set job_alerts_enabled = true. Trigger: when new job matches talent's skills + location, send push notification. 5 Apply Talent taps Apply on a job post. Status: APPLIED. Code: POST /applications {jobId, talentId}. Status=APPLIED. 6 Employer Action Employer views applicants list. Selects for Interview or Rejects. Code: PATCH /applications/{id} {status: 'INTERVIEW' | 'REJECTED'}. 7 Status Flow States: APPLIED → INTERVIEW → SELECTED | REJECTED. Talent sees live updates. Code: WebSocket or polling for status changes. Push notification on each state change. 8 No-Show Control If talent accepts interview but doesn't attend: system adds negative mark. Repeated = lower search priority. Code: Track interview_accepted + attended. If accepted but !attended: increment no_show_count. Apply rank penalty in search query. Current Flow Issue — Find a Job The negative behaviour control (no-show tracking) is not yet linked to the search ranking algorithm. Currently, a talent who repeatedly skips interviews still appears at the top of employer search results. The ranking query must factor in no_show_count as a penalty weight. 5. Earn by Working (Live Worker Engine) The real-time gig earning system. Workers register skills, get verified, set rates, go live, and earn hourly — without any middlemen. Step Action Description / Code Requirement 1 Skill Registration Worker adds skills: Electrician, Driver, Cleaner, Nurse, etc. Each skill requires individual verification. Code: POST /worker/skills {skill, documents[]}. Status: VERIFICATION_PENDING. 2 Admin Verification Admin reviews proof documents and approves. Code: Admin panel PATCH /admin/skills/{id}/approve. Webhook triggers skill activation. 3 Rate Setting Worker sets hourly rate within system range. Can update max 2 times per day. Code: Store rate_updated_count per day. Reset at midnight. Reject if count >= 2. 4 Availability Schedule Worker sets weekly schedule: Mon 9am-5pm, Tue 3pm-7pm, etc. Code: Store availability_schedule as JSON. Use to filter workers during customer search. 5 Go Live Toggle Worker toggles Active. System checks: wallet_balance >= 12 (except first job). Code: GET /worker/wallet. If balance < 12 AND has_done_first_job: show 'Top up Rs.12 to go live'. Else allow. 6 Job Notification Customer triggers search. Matching workers receive push notification. Code: FCM push with job details. Worker has 90 seconds to accept. 7 Accept Job Worker taps Accept. First to accept gets the job. Code: Atomic DB transaction — set job.worker_id if null. Return conflict error to slower acceptors. 8 Arrival & OTP Worker arrives, taps Arrived. OTP sent to on-site contact. Worker enters OTP. Code: Generate 6-digit OTP, store with 5-min TTL. SMS/WhatsApp to contact number. 9 Completion & Payment Worker taps Finish. Second OTP. Customer confirms. Payment calculated. Code: Billing = max(actual_hours, 1) * hourly_rate. Process payment. 10 Rs.12 Bonus After first job: credit Rs.12 to worker wallet. Code: Check is_first_job flag. Credit wallet, set is_first_job = false. Strike System Worker gets a strike for: accepting job then not moving (auto-cancel), or repeated no-shows. Implement worker_strikes table. After 3 strikes: suspend account. Rate Limit Rule Hourly rate update: max 2 changes per day. Enforce via rate_update_log table with date + count. Reset count at midnight IST. Wallet Logic Minimum Rs.12 balance required to go live (after first job). Deduct Rs.12 on each job acceptance. Refund if job cancelled before arrival. 6. List My Business Module Allows service providers and companies to register and manage their offerings. Three business categories, each with independent unit management. 6.1 Service Business (Hall / Catering / Decoration / Photography) Unit-Based Model: One owner can manage multiple independent business units of the same type. Step Action Description / Code Requirement 1 Select Category User selects: Service Business → Hall/Catering/Decoration/Photography. Code: Route to service type selector. 2 First-Time Check System checks: does this user already have units of this type? Code: GET /businesses?ownerId={uid}&type={type}. If none: show Add Unit form. 3 Register Unit User fills Company Name, Location, Description, uploads proof. Status: VERIFICATION_PENDING. Code: POST /businesses {type, name, location, description, documents[]}. 4 Admin Approval Admin verifies and approves. Code: PATCH /admin/businesses/{id}/approve. Status: VERIFIED. Push notification to owner. 5 Dashboard User sees Unit Cards — one card per business. Plus Add New button. Code: GET /businesses?ownerId={uid}&type={type}. Render card grid. 6 Unit Management Clicking a card: Manage Calendar, Add Capacity, Add Offline Days, Edit Details, Delete, Transfer Ownership, Add Operators. Code: Each action = separate API endpoint on /businesses/{id}/*. 7 Operator Add Owner adds Manager. Manager can approve bookings and manage availability. Cannot transfer ownership. Code: POST /businesses/{id}/operators {userId, role:'manager'}. Permission middleware checks role. 8 Ownership Transfer Owner initiates transfer. OTP required from both parties. 24-hour hold applied. Code: POST /businesses/{id}/transfer {newOwnerId}. OTP verification flow. Cron job lifts hold after 24hrs. 6.2 Hiring Business Step Action Description / Code Requirement 1 Register Company Company Name, Business Type, Location, GST/Legal proof. Status: VERIFICATION_PENDING. Code: POST /hiring-businesses {name, type, location, documents[]}. 2 Create Hiring Units Owner creates department units (e.g., Development Team, Support Team). Each hires independently. Code: POST /hiring-businesses/{id}/units {name, description}. 3 Add Operators (HR) Owner adds HR managers as Operators. HR can post jobs, view applicants, move candidates. Cannot transfer ownership. Code: POST /hiring-businesses/{id}/operators {userId, role:'hr'}. 4 Post Job HR posts role with: title, experience, salary, location, skills required. Code: POST /jobs {businessId, unitId, title, experience, salary, location, skills[]}. 5 Talent Match On job post: system queries talent_profiles where skills overlap AND preferred_locations match. Send push alert. Code: Background job after job creation. Batch push via FCM. 6 Pipeline Management HR moves candidates: APPLIED → INTERVIEW → SELECTED / REJECTED. Code: PATCH /applications/{id} {status, note?}. Each state change notifies talent. 7 Ownership Transfer Same 24-hour OTP hold as Service Business. Code: Shared ownership transfer service. 6.3 Rental Business Step Action Description / Code Requirement 1 Register Vendor Vendor lists equipment: Generator, Mic Set, Cooler, Pressure Washer, etc. Code: POST /rental-items {vendorId, name, category, specs, hourlyRate, minHours, photos[]}. 2 Calendar Setup Each item has its own calendar. Vendor sets availability. Code: GET/POST /rental-items/{id}/calendar. 3 Receive Requests Customer sends rental request (with optional note). Vendor reviews. Code: GET /rental-bookings?vendorId={id}&status=REQUESTED. 4 Approve/Reject Vendor approves → customer confirms → day blocked. Code: PATCH /rental-bookings/{id}/status {action: 'approve'|'reject'}. On confirm: block date in calendar. 7. Current Flow Issues & Fixes Required The following issues have been identified across all modules based on the intended system design vs. current implementation state: # Flow / Module Current Issue Fix Required 1 Book Services — Location No separation between user's current location and event location. Both Plan Services and Instant Help show user's GPS as default for everything. Implement two distinct location capture flows: selectedEventLocation for Plan Services; serviceLocation (GPS or manual) for Instant Help. 2 Instant Help — On-Site Contact Presence Check screen not implemented. No way to add an on-site contact person. Arrival OTP always goes to booker — wrong if someone else is on-site. Build Presence Check UI: 'I will be present' / 'Someone else'. If someone else: Name + Relation + Phone form. Store as on_site_contact in booking. Route OTP to this number. 3 Instant Help — Movement Monitor No background GPS movement check implemented after worker accepts. Worker can accept and stay stationary with no consequence. Implement background job: 7-min push reminder, 10-min GPS check. If no movement detected: auto-cancel job, add worker strike. Use scheduled task + GPS API. 4 Plan Services — Multi-vendor Currently only single booking requests are supported. System does not allow a customer to send the same booking request to multiple vendors simultaneously. Allow parallel booking requests to multiple vendors. Status: REQUESTED for each. First vendor to approve + customer confirms = CONFIRMED. Cancel others automatically. 5 Find a Job — No-Show Ranking No-show tracking exists but is not linked to talent search ranking. Repeated no-show talent still appears at top of employer results. Add penalty weight to talent search query. Deduct search_rank score based on no_show_count. Weight formula: base_score - (no_show_count * penalty_factor). 6 Earn by Working — Rs.12 Wallet Rs.12 activation requirement exists in spec but wallet balance check before going live is not implemented. Worker can go live with zero balance. Before allowing Active toggle: if has_done_first_job == true, check wallet_balance >= 12. Block toggle and show top-up prompt if insufficient. 7 Earn by Working — Rate Limit Workers can update their hourly rate unlimited times per day. No daily rate-change cap enforced. Log each rate update in rate_update_log with timestamp. Count updates for today. Reject if count >= 2 with message: 'Rate can only be updated twice per day.' 8 List My Business — Ownership Transfer Hold 24-hour hold after ownership transfer is designed but the cron/scheduled job to lift the hold and finalise ownership is not implemented. Implement scheduled job: 24hrs after transfer initiation, verify both OTPs were confirmed, then set new_owner as owner. If not confirmed within 48hrs, expire the request. 9 List My Business — Operator Permissions No permission middleware enforces what operators can/cannot do. An operator can currently access ownership transfer endpoints. Add role-based permission middleware. Check user role per business unit before each action. Operators: bookings + calendar only. Owners: all actions including transfer. 10 All Modules — Push Notifications Push notification triggers (job alert, booking status, application status) are not connected to FCM/APNs. Status changes update DB but no notification is sent. Integrate Firebase Cloud Messaging (FCM). Create notification service. Trigger pushes on: booking status change, job request received, application state change, OTP delivery. 8. Common Principles Across All Modules Safe Harbor Gixbee never assigns. Every booking, acceptance, and hiring decision is made by the user. System only facilitates matching and communication. Intent-Based Design A user is not locked into a single role. The same account can book services, work as a gig worker, and list a business simultaneously. Location-Driven Filtering All service discovery is location-first. Plan Services: event location. Instant Help: service location. Jobs: preferred work location. OTP Verification Two OTP gates in Live Worker flow: Arrival OTP (work begins) and Completion OTP (work ends). Both sent to on-site contact, not necessarily the booker. Calendar Block Confirmed bookings (Hall, Catering, Decoration, Photography, Rental) block the calendar immediately. Prevents double-booking. Multi-Unit Ownership One user can own multiple businesses of the same or different types. Each unit is independent — separate calendar, operators, and management. Operator vs Owner Operators (Managers, HR) can manage day-to-day work but cannot transfer ownership. Ownership remains with the registered owner until OTP-verified transfer. Verification Gate Skills (workers), businesses, and rental items all require admin approval before going live. Status flow: PENDING → VERIFIED / REJECTED. 10.
