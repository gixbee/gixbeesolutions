## 🔴 Critical — Secrets & API Keys

These must be fixed before any public release or repository sharing.

---

### 1. Supabase URL and Anon Key — `lib/main.dart` (lines 18–19)

**Found:**
```dart
await Supabase.initialize(
  url: 'https://your-project-id.supabase.co',
  anonKey: 'your-anon-key',
);
```

**Problem:**  
Supabase credentials are hardcoded directly in `main()`. Even placeholder values train bad habits and break environment separation between dev, staging, and production.

**Fix:**  
Use `--dart-define` at build time and read via `String.fromEnvironment`:

```dart
await Supabase.initialize(
  url: const String.fromEnvironment('SUPABASE_URL'),
  anonKey: const String.fromEnvironment('SUPABASE_ANON_KEY'),
);
```

Run the app with:
```bash
flutter run \
  --dart-define=SUPABASE_URL=https://your-project.supabase.co \
  --dart-define=SUPABASE_ANON_KEY=your-anon-key
```

### 2. Live Firebase API Keys — `lib/firebase_options.dart` (lines 36–37, 43–44, 50–51)

**Found:**
```dart
static const FirebaseOptions web = FirebaseOptions(
  apiKey: 'AIzaSyDPz5C2KRIv8luzIpO8JoPTkpizHg1XxEQ',  // REAL KEY
  appId: '1:157133477367:web:732fb12a756d0176a14af2',
  ...
);

static const FirebaseOptions android = FirebaseOptions(
  apiKey: 'AIzaSyCAUSU1ZG8LCgLYnv2BSR7gZIH8asGkqg4',  // REAL KEY
  ...
);
```

**Problem:**  
Real Firebase API keys are committed to source control. Anyone with access to the repository can misuse these keys. This is a security breach if the repo is ever made public.

**Fix:**  
1. Add `firebase_options.dart` to `.gitignore`
2. Use FlutterFire CLI to regenerate per environment:
```bash
flutterfire configure --project=gixbee-dev   # for dev
flutterfire configure --project=gixbee-prod  # for prod
```
3. Use CI/CD secrets to inject the correct file at build time.

Also note: the iOS `appId` is still a placeholder:
```dart
appId: '1:157133477367:ios:YOUR_IOS_APP_ID',  // never filled in
```
iOS Firebase integration will silently fail until this is set.

### 5. Android Emulator Host IP — `lib/core/config/app_config.dart` (lines 12, 22)

**Found:**
```dart
defaultValue: 'http://10.0.2.2:3000',
```

**Problem:**  
`10.0.2.2` is the Android emulator's alias for the host machine's localhost. This IP does not work on real Android devices, physical test devices, or CI runners.

**Fix:**  
Always pass `API_BASE_URL` explicitly via `--dart-define` for every real device build and CI pipeline. Never rely on the emulator IP as a defaultValue in production builds.

### 6. External Avatar Fallback URL — `lib/features/profile/profile_screen.dart` (~line 55)

**Found:**
```dart
backgroundImage: NetworkImage(user?.avatar ??
  'https://ui-avatars.com/api/?name=${user?.name ?? "User"}&background=6200EE&color=fff'),
```

**Problem:**  
A third-party service (ui-avatars.com) is hardcoded as a fallback. If the service goes down, changes its URL format, or imposes rate limits, all users without avatars will show broken images. There is no way to change it without a code release.

**Fix:**  
```dart
// In AppConfig:
static const String avatarBaseUrl = 'https://ui-avatars.com/api/';

// In profile_screen.dart:
'${AppConfig.avatarBaseUrl}?name=${user?.name ?? "User"}&background=6200EE&color=fff'
```

### 30. Booking Status as Inline Strings — `lib/features/booking/waiting_for_worker_screen.dart`

**Found:**
```dart
if (status == 'ACCEPTED') { ... }
else if (status == 'CANCELLED' || status == 'REJECTED') { ... }
```
Status values are inline strings compared directly. A single typo silently breaks the entire booking flow.

**Fix:** Create an enum:
```dart
enum BookingStatus { accepted, cancelled, rejected, pending }

extension BookingStatusX on String {
  BookingStatus toBookingStatus() => BookingStatus.values.firstWhere(
    (e) => e.name.toUpperCase() == toUpperCase(),
    orElse: () => BookingStatus.pending,
  );
}
```
